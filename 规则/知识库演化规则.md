# 知识库演化规则

知识库的结构和规则本身也是暂时的假设。它们可以随真实使用而改变，但改变的目的只能是减少捕捉、查找、连接、核验和写作的摩擦，而不是让目录看起来更复杂、更完整。

> 规则只因反复出现的真实摩擦而改变；先改变未来的新内容，不急于回填历史内容；任何结构性改变都必须可说明、可验证、可回滚。

## 1. 不随试验改变的底线

- 不静默改写用户原话、引文和原始材料。
- 除非用户明确指定永久删除的对象与范围，不删除任何用户材料；只允许移动、重命名、建立替代入口或归档。
- 不把 AI 的推演、转述或补写自动归为用户已经认领的思想。
- 不混淆事实、来源转述、解释、个人判断、推测和开放问题。
- 不为适应新规则而删除仍有认识价值的旧内容。
- 不因一次例外或理论上的整洁感批量移动历史文件。
- 未经明确采纳的提案不是正式规则。

涉及原始材料损坏、来源丢失、隐私泄漏或大量断链时，应立即停止相关试验并修复，不等待重复次数。

## 2. 三种变更等级

### 日常维护

错字、明确断链、遗漏索引、非法元数据和已经确认的来源定位可以直接修复，但仍应保留用户原意。

### 工作流试验

模板章节、导航入口、复盘问题和局部命名方式应先小范围试用。一次只运行一个试验，默认持续未来 3 次真实使用或最多两个复盘周期。

### 结构变更

目录层级、类别新增或退役、元数据允许值、批量迁移、自动化脚本和跨库同步会影响大量内容，必须由用户明确采纳，并事先写明影响范围、材料迁移映射与回滚方法。

## 3. 类别的新增、拆分、合并与退役

这里的“类别”包括活动目录、索引入口、主题地图、稳定 `domain` 值及其他承担分类职责的导航条目。类别是可演化的结构，不是用户材料本身。

### 允许的变化

- **新增**：现有类别持续容纳不了真实内容，或新领域已形成独立检索入口。
- **拆分**：一个类别内部已经出现稳定子群，且浏览或检索确实困难。
- **合并**：两个类别边界长期重叠，用户每次分类都需要在两者之间反复判断。
- **退役**：类别已无活动用途、被更清晰的结构替代，或连续两个复盘周期都不能改善查找。

内容数量只是信号。新增独立领域通常应已有至少 5 条实质内容；减少类别也不能只因条目少，而要确认它不再提供独立的理解或检索价值。用户明确提出结构需求时，可以直接进入提案与确认，不必机械等待次数。

### 类别变更闭环

1. **盘点**：列出类别内全部笔记、原始材料、附件、索引、地图入口、元数据值和外部回链。
2. **映射**：使用[类别变更提案模板](../模板/类别变更提案.md)写清旧类别到新类别的逐项去向，以及哪些内容保持不动。
3. **确认**：类别变化属于结构变更；执行前由用户明确采纳方案。确认“合并类别”不等于授权删除其中材料。
4. **迁移**：只移动或重命名材料，随后更新相对链接、`domain`、索引、地图和必要的 `MEMORY.md` 入口。原始材料的原文区不改写。
5. **退役入口**：旧类别从活动导航移除后，将其索引或说明移入 `90-归档` 作为结构历史；必要时在旧位置保留指向新位置的过渡说明。
6. **验证**：全文搜索旧类别名和旧路径，确认零断链、零遗漏、零孤立材料，并抽查迁移前后文件数量和来源链。
7. **回滚**：按映射把材料移回原位置，恢复旧索引和元数据；回滚不得删除迁移期间新增的用户内容。

“减少类别”只表示从活动分类和导航中退役。只要文件包含用户原话、用户提供的附件、用户撰写或修改的内容、由用户材料形成的卡片，或仍有认识价值的历史，就属于用户材料，不能以“清理空类别、去重、合并、重构、发布或回滚”为理由删除。

AI 生成且从未承载用户内容的临时文件、缓存和可再生成产物不属于用户材料，可以清理；不能用这一例外删除含有用户输入的笔记或附件。用户若要求永久删除，必须明确到具体对象或可核对范围；仅说“整理、精简、合并、减少类别”不构成删除授权。

## 4. 何时值得提出改进

以下是观察信号，不是必须追求的指标：

| 现象 | 建议触发条件 |
| --- | --- |
| 同一种工作流摩擦 | 出现 3 次，或连续两个周复盘出现 |
| 明知某笔记存在却找不到 | 一个月内发生 2 次 |
| 模板缺少同一信息 | 至少 3 篇笔记都临时补写 |
| 模板字段无用 | 最近 5 次使用都为空或被跳过 |
| 分类需要同一种例外 | 累计 3 条真实内容 |
| 新领域需要独立入口 | 至少已有 5 条实质内容，且现有地图已造成查找困难 |
| 目录需要拆分 | 已发生实际浏览困难；文件数量本身不是理由 |
| 重要笔记长期孤立 | 连续两个复盘周期无连接，且确有复用价值 |
| 试验到期 | 到达两个复盘节点仍未评估时，必须作出决定 |

断链、非法元数据或明确的索引遗漏属于日常维护，不必为了修复它们建立提案。

## 5. 演化闭环

### 第一步：记录摩擦

先把问题写入[改进记录](../80-复盘/改进记录.md)或当周复盘，附上真实例子。首次、低影响的偶发问题默认只观察。

### 第二步：提出最小假设

达到触发条件后，复制[改进提案模板](../模板/改进提案.md)。一个提案只解决一种摩擦，并明确：

- 改什么；
- 不改什么；
- 怎样知道它有效；
- 如果无效怎样恢复。

### 第三步：建立基线

只记录与问题直接有关的简单事实，例如“最近 5 次使用中这个字段 5 次为空”或“两次寻找已知笔记都超过两分钟”。没有基线就无法区分真实改善和主观新鲜感。

### 第四步：小范围试验

- 模板变化只用于之后 3 次新笔记，默认不回填旧文件。
- 导航变化先改一张地图或一个索引。
- 目录和元数据变化先在 3—5 个样本上验证，并取得用户确认。
- 试验不能修改原始材料的原文区，也不能损害来源链和用户表达。

### 第五步：评估

至少回答：

- 原来的摩擦是否减少？
- 是否增加了填写、维护或查找成本？
- 是否破坏来源、链接、声音归属或历史可读性？
- 是否发生过足够的真实使用？没有使用不能算验证成功。

### 第六步：作出决定

- **采纳**：真实使用显示有效，且未触碰保护底线。
- **调整后再试**：问题成立，但方案过大或设计不合适；最多调整一次。
- **回滚**：没有改善、代价更高或产生来源与链接风险。
- **放弃**：摩擦没有再次发生、场景消失或证据不足。

提案由 AI 起草时，AI 不得自行把它判定为用户已经采纳。

提案沿用现有 `status`：`seed` 表示已提出但未试验，`growing` 表示试验中，`completed` 表示已经评估并作出决定；被放弃或回滚的提案仍保留决定记录，不以删除文件代替结论。

### 第七步：同步或回滚

采纳后在同一次变更中同步正式规则、模板、索引和必要的 `README.md`、`AGENTS.md`。默认只对未来内容生效，历史迁移另行决定。

回滚时恢复试验涉及的规则或模板，但保留提案和评估记录；试验期间新增的用户内容不得因此丢失。

## 6. 每月抽样的健康信号

不建设分数体系，只做少量抽样：

1. **可找回性**：随机选择 3 条近期笔记，能否在两分钟内通过标题、别名、地图或关键词找到。
2. **入口有效性**：首页当前焦点是否真实，活跃主题是否从地图或索引可达。
3. **捕捉滞留**：收件箱最老条目的日期；超过 30 天是否有保留原因或下一步。
4. **连接质量**：抽样的 `growing`、`evergreen` 笔记是否有一个有意义的连接或明确的孤立原因。
5. **来源可回溯**：抽样 3 条依赖外部事实的笔记，来源能否定位；个人判断是否被误当成事实。
6. **原始材料完整性**：新增材料是否记录来源、范围、完整性、敏感性和核验状态；派生笔记能否回到原始材料。
7. **结构一致性**：活动链接无断链，元数据没有未经规则允许的新值。
8. **停止生长的内容**：超过 90 天未更新的 `draft` 或 `growing` 是否仍有明确下一步。
9. **维护负担**：模板和分类是否开始妨碍捕捉；若整理成本超过复用收益，优先删减规则。
10. **演化闭环**：当前试验不超过一个；到期提案已有明确决定。
11. **类别有效性**：新增类别确有独立用途；待合并或退役的类别已有完整迁移映射，而不是只看条目数量。
12. **材料保护**：结构调整前后，用户材料数量、来源链和附件定位可核对；不存在未经用户明确授权的删除。

这些信号用于诊断，不是绩效目标。不要为了让数字好看而制造链接、强行归档或机械更新日期。
