# 知识库规则

## 1. 内容生命周期

每条内容不必走完所有阶段，但应知道自己处在哪里。

1. **捕捉**：没有整理的闪念、问题或材料，放入 `00-收件箱`。
2. **澄清**：判断它是个人思考、百科知识、资料、分析任务还是写作素材。
3. **连接**：至少链接到一个已有条目或主题地图；暂时无连接时写明“孤立原因”。
4. **沉淀**：补足背景、论证、反例、来源和相关条目，使其能脱离当时语境被理解。
5. **产出**：在分析或写作中组合多张卡片，形成更完整的表达。
6. **归档**：失效、重复、被合并或不再活跃的内容移入 [`90-归档`](../90-归档/README.md)。除非用户明确要求永久删除具体材料，否则用户材料只能移动或归档，不能删除。

只要用户内容要成为正式笔记，具体执行顺序以[记录、拆分与关联工作流](记录与关联工作流.md)为准：原始保存先行，拆分后逐条搜索并建立有语义的链接。

## 2. 放在哪里

- 内容尚不清楚：`00-收件箱`
- 重点是“我的想法、判断或问题”：`10-思考`
- 重点是“它是什么，并且可供反复查阅”：`20-百科`
- 重点是“我从某个来源获得了什么”：`30-资料`
- 重点是“为一个问题综合证据并得出结论”：`40-分析`
- 重点是“面向读者形成作品”：`50-写作`
- 重点是“导航一组相关内容”：`60-地图`
- 内容已停止生长或被取代：`90-归档`
- 重点是“跨多次任务持续有用的学术脉络与进展”：根目录 `MEMORY.md`

一条笔记只能有一个主要住所。跨领域关系通过元数据和链接表达，不复制文件。

类别本身可以新增、拆分、合并或退役；减少类别只改变活动结构和导航，不删除其中材料。具体盘点、迁移、验证和回滚遵循[知识库演化规则中的类别变更闭环](知识库演化规则.md)。

### 原始材料的边界

`30-资料/原始材料` 保存尚未被摘要、解释或改写的原始输入，例如用户原话、对话记录、访谈、截图转写、观察记录和文档快照。

- 原文区保持原始顺序和措辞，不静默纠错、润色或补全。
- 缺失、截断和无法辨认的内容必须显式标记。
- 来源、取得日期、保存范围和完整性必须可追溯。
- 原始材料可使用 `capture_scope`（`full` / `excerpt`）、`integrity`（`intact` / `partial` / `unknown`）和 `sensitivity`（`private` / `restricted` / `public`）描述保存状态；这些字段不改变原文内容。
- `status: completed` 只表示“本次保存记录已完成”，不表示原始来源完整；来源完整性以 `capture_scope`、`integrity` 和正文说明为准。
- 对原始材料的概括放在 `30-资料` 的其他笔记中；独立判断放在 `10-思考`；综合论证放在 `40-分析`。
- 只要正式笔记使用了用户输入，就必须先有原始材料记录。派生笔记必须链接回原始材料，原始材料也必须回链派生笔记；不得用整理后的文字覆盖唯一的原始记录。
- 一次输入或连续会话可以只保存一份原始记录，并派生多张卡片。短文字使用[轻量原始记录模板](../模板/轻量原始记录.md)；大文件或附件只保存一份原件，再以 Markdown 记录文件位置、范围、完整性、敏感性和核验状态。
- 外部 URL 本身不等于已经保存。无法完整保存时，保留合法节选或准确定位，明确标记 `excerpt` / `partial`，不得用摘要冒充原文。
- 原始附件保持原文件，不以压缩、转码或 OCR 文本替代；另建同名或相邻 Markdown 记录说明来源、核验方式和派生关系。
- 原始材料默认按私密内容处理。公开、同步或用于写作前，单独检查个人信息、版权和引用范围；Git 版本历史不能代替异地备份。

## 3. 文件命名

- 使用能直接表达内容的中文标题，例如 `技术债是一种时间上的权衡.md`。
- 避免 `一些想法.md`、`随笔2.md`、`未命名.md` 等无法检索的名称。
- 收件箱或复盘可使用日期前缀：`2026-07-16-关于自由意志的疑问.md`。
- 百科条目使用稳定名称：`存在主义.md`、`汉娜·阿伦特.md`、`百年孤独.md`。
- 同名时在标题后加限定语：`状态（软件系统）.md`、`状态（哲学）.md`。
- 文件移动后及时修正所有相对链接。

## 4. 一条笔记的边界

- 一条笔记围绕一个中心命题、概念、人物、作品或问题。
- 标题应尽量是一句可判断、可搜索的话；百科实体可直接使用规范名称。
- 如果正文中出现可独立讨论且会被多次引用的内容，把它拆成新笔记并建立双向链接。
- 短不是问题；含混和不可复用才是问题。
- 搜索发现已有同一中心命题时，优先把新原始材料并入现有笔记并记录修订；只有立场确有差异时才另建卡片并说明“修订”或“张力”。

### 思考卡片的声音顺序

思考卡片固定按“用户原始表达 → 用户观点 → AI 辅助 → 待确认 → 仓库关联”组织。

- 用户观点是主体；AI 只提供澄清、概念候选、反例、推论和来源线索。
- 用户只有问题时写“观点尚未形成”，不得由 AI 代写结论。
- 用户确认核心命题不等于确认整段 AI 展开。
- AI 的独立长论证另放 `40-分析` 并标明归属，不塞进思考卡主体。

## 5. 元数据

正式笔记使用 YAML 头部，字段保持克制：

```yaml
---
title: 标题
aliases: []
type: thought
domain: [哲学]
status: seed
created: 2026-07-16
updated: 2026-07-16
tags: []
related: []
sources: []
---
```

### 允许值

- `type`：`inbox`、`thought`、`concept`、`person`、`work`、`theory`、`case`、`source`、`analysis`、`writing`、`map`、`review`
- `domain`：优先使用 `哲学`、`软件架构`、`文学`、`跨领域`，需要时再增加稳定领域
- `status`：
  - `seed`：只有种子，尚未展开
  - `growing`：正在补充和建立连接
  - `evergreen`：内容相对稳定，仍允许修订
  - `draft`：面向产出的草稿
  - `completed`：本阶段已经完成
  - `archived`：已归档

`tags` 只表达跨目录、跨领域且会反复使用的特征。不要把文件夹、类型和领域再次做成标签。

`related` 保存规范标题，供人阅读和检索，不作为唯一链接依据。重命名时需全文检索并同步更新；真正的关系仍应在正文中用可验证的 Markdown 链接表达。

## 6. 链接与索引

- 默认使用标准 Markdown 相对链接，如 `[理论与流派索引](../20-百科/理论与流派/_索引.md)`，保证工具无关。
- 新笔记拆分完成后，逐条搜索文件名、标题、别名、标签、正文、索引、地图和 `MEMORY.md`。至少搜索用户独特原句、核心概念及同义词、上位或相反问题三类线索。
- 只有能说明为“支持、扩展、修正、冲突、前提、反例、例证、应用、同源、类比或同一问题”时才建立关系；共同出现关键词不足以证明相关。
- 正文中的链接应说明关系，不只在末尾堆砌“相关链接”。
- 主题地图负责人工策展，不追求自动罗列全部文件。
- 重要关系可以在相关笔记中互相链接；不强求每条链接机械对称。
- 未找到实质关联时，不制造链接。记录检索日期、范围、关键词、结果、孤立原因和下次重查条件；领域索引只能标作导航入口，不能冒充语义关系。

## 7. 长期学术记忆

根目录 [`MEMORY.md`](../MEMORY.md) 是默认私密的策展层，用于保存跨多次任务仍然有用的学术方向、用户已确认命题、思想修订、研究里程碑、当前焦点和长期问题。

- 它不是每日日志、原始材料副本、全库索引或事实证据。
- 只有用户确认可长期复用的命题、明确修订或否定旧命题、研究焦点或方法发生变化、完成重要里程碑，或用户明确要求时才更新。
- 主体区只放用户已确认内容；AI 候选只能作为“正在发展”的待确认线索。
- 每条记忆用简短摘要链接到正式笔记及其原始材料；保留修订时间线，不静默覆盖旧立场。
- 普通捕捉、一次性闪念和细碎处理进度不写入。

## 8. 修改与历史

- 不覆盖仍有价值的原始思路；重大转变可增加“修订记录”说明为何改变。
- 合并笔记时，在旧文件中留下新位置后再归档，避免链接突然失效。
- 用户材料包括用户原话、附件、用户撰写或修改的内容、由用户材料形成的卡片，以及仍有认识价值的历史。除非用户明确指定永久删除的对象与范围，助手只能移动、重命名、合并入口或归档这些材料。
- “整理、精简、去重、合并、重构、减少类别、回滚或生成公开版”都不构成删除授权。重复内容也保留旧入口或移入归档。
- AI 临时文件、缓存和可再生成产物只有在从未承载用户内容时才可清理。
- 每次集中整理后提交一次版本记录，提交说明写清“为什么改变”，不只写“更新”。

## 9. 维护节奏

- **随时**：把想法投入收件箱。
- **每周**：清理收件箱、更新当前焦点、复查近期关联搜索、记录未决问题。
- **每月**：检查主题地图和长期记忆，合并重复条目，标记长期孤立或停止生长的内容。
- **按需**：当某一主题积累到难以浏览时，再增加子目录；不提前设计空分类。

## 10. 规则如何演化

- 明确断链、非法元数据和遗漏索引可以直接修复。
- 重复出现的摩擦先进入[改进记录](../80-复盘/改进记录.md)，不要因一次例外增加规则。
- 模板、导航和流程变化先做小范围试验；同一时间只运行一个试验。
- 目录、元数据允许值、批量迁移和自动化属于结构变更，必须由用户明确采纳。
- 类别变更使用[类别变更提案模板](../模板/类别变更提案.md)，先盘点和映射，再迁移与验证；不得把“类别退役”实现成材料删除。
- 指标只是诊断信号，不是需要追求的绩效数字。

内容处理完整流程见[记录、拆分与关联工作流](记录与关联工作流.md)；规则演化完整流程见[知识库演化规则](知识库演化规则.md)。

## 11. 公开副本

- 私库中的 `sensitivity: private` 内容、真实长期记忆和个人使用审计默认不公开。
- 公开空白版本必须从私库生成净化副本，不在原仓库中删除私人材料来“制造公开版”。
- 具体文件排除、共享导航净化、敏感词反查、断链检查与校验文件遵循[发布与隐私规则](发布与隐私规则.md)。
